# 개인정보 처리 설계 (cafe-os)

## 설계 철학: 개인정보를 안 받는 게 최고의 보안

cafe-os 는 의도적으로 **개인을 식별할 수 있는 정보를 수집하지 않는다.**
수집하지 않으면 보호할 의무도, 유출 위험도, 과태료 리스크도 없다.

---

## 1. 체류시간 — 추적하지 않음

- IP 주소로 체류시간을 추적하지 **않는다.**
- 기록하는 것은 **주문 시각**(`ordered_at`)뿐이며, 이는 매출·수요 분석용이다.
- 이유:
  - IP는 한국 개인정보보호법상 **개인정보**로 취급된다.
  - 공유기가 IP를 손님마다 돌려쓰므로(DHCP) 체류시간 추적은 부정확하다.
  - "언제 얼마나 팔리나"는 주문 데이터로 충분히 분석된다.

## 2. 재방문/쿠폰 — 익명 코드 방식

- 휴대폰번호·이름·이메일을 **받지 않는다.**
- 대신 무작위 **쿠폰코드**(예: `T5W4-A4VN`)로 스탬프를 적립한다.
- 카페는 코드 소유자가 누구인지 **알 수 없다.**
- 손님이 코드를 직접 보관(스크린샷). 분실 시 복구 불가 — 익명성의 대가.

### 법적 효과
이 데이터는 특정 개인을 알아볼 수 없으므로 **개인정보에 해당하지 않는다.**
따라서 다음 의무에서 자유롭다:
- 수집·이용 동의 절차
- 안전성 확보조치(암호화 등)
- 보유기간 후 파기
- 유출 시 신고·통지

> ⚠️ **절대 하지 말 것**: `coupons` 테이블에 phone, name, email 등
> 개인식별 컬럼을 추가하는 순간, 카페 사장은 '개인정보처리자'가 되어
> 위 모든 의무를 지게 됩니다. 코드에 검증 테스트로 막아두었습니다.

## 3. 만약 폰번호 멤버십이 꼭 필요하다면

별도 옵트인 모듈로 분리하고 반드시:
1. 수집 항목·목적·보유기간 **고지 후 동의** 체크박스 (필수)
2. 저장 시 **암호화**
3. 마케팅 문자는 **별도 동의** (정보통신망법 — 무단 발송 시 과태료)
4. 손님의 **열람·삭제 요청** 처리 기능

→ 소규모 카페엔 권장하지 않음. 익명 쿠폰으로 충분.

## 4. 손님 화면 고지 문구 (예시)

주문 페이지 하단에 표시 권장:
```
본 주문 시스템은 휴대폰번호 등 개인정보를 수집하지 않습니다.
쿠폰은 익명 코드로 발급되며, 적립 내역만 기록됩니다.
```

## 5. 수집/미수집 정리표

| 항목 | 수집? | 용도 |
|------|:----:|------|
| 주문 메뉴·금액·시각 | ✅ | 매출·수요 분석 |
| 날씨 | ✅ | 수요 영향분석 |
| 테이블 번호 | ✅ | 주문 전달 |
| 쿠폰코드(익명) | ✅ | 스탬프 적립 |
| 휴대폰번호 | ❌ | — |
| 이름·이메일 | ❌ | — |
| IP 기반 체류시간 | ❌ | — |
| 손님 위치·기기정보 | ❌ | — |
